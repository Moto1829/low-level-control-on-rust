# 第7章: 高性能データ構造

## 概要

Rustでは、適切なデータ構造の選択がプログラムのパフォーマンスに直結します。同じアルゴリズムでも、データのメモリ上のレイアウトが異なるだけで、実行速度が数倍から数十倍変わることがあります。本章では、現代のハードウェア（特にCPUキャッシュ）を意識した高性能データ構造の設計と実装を学びます。

## なぜデータ構造の選択がパフォーマンスに直結するか

### メモリ階層とアクセスコスト

現代のCPUは、計算速度とメモリ速度の間に大きなギャップ（「メモリウォール」）があります。このギャップを埋めるために、複数段のキャッシュが使用されています。

```
CPU Register  : ~0.3 ns   (1 サイクル)
L1 Cache      : ~1 ns     (4 サイクル)      32 KB
L2 Cache      : ~4 ns     (12 サイクル)     256 KB
L3 Cache      : ~40 ns    (40 サイクル)     8-32 MB
DRAM (Main)   : ~100 ns   (200+ サイクル)
NVMe SSD      : ~100 µs
HDD           : ~10 ms
```

```
┌─────────────────────────────────────────────────────────────┐
│                        CPU Chip                             │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │  Core 0  │  │  Core 1  │  │  Core 2  │  │  Core 3  │  │
│  │ L1 32KB  │  │ L1 32KB  │  │ L1 32KB  │  │ L1 32KB  │  │
│  │ L2 256KB │  │ L2 256KB │  │ L2 256KB │  │ L2 256KB │  │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘  │
│       └──────────────┴─────────────┴──────────────┘        │

                        └────────────┘
```

### キャッシュフレンドリーなコードの重要性

メモリへのアクセスパターンが「予測可能」かつ「連続的」であれば、CPUのプリフェッチャーが事前にデータをキャッシュに読み込み、待ち時間をほぼゼロにできます。逆に、ポインタをたどるような「ランダムアクセス」パターンはキャッシュミスを多発させ、性能を大幅に低下させます。

```
// キャッシュフレンドリー: 連続メモリアクセス
let vec: Vec<i32> = (0..1000000).collect();
let sum: i32 = vec.iter().sum(); // 高速 - プリフェッチが効く

// キャッシュアンフレンドリー: ポインタチェーシング
struct Node { value: i32, next: Option<Box<Node>> }
// 各ノードがヒープ上の別の場所に存在 -> キャッシュミス多発
```

### データ構造選択の判断基準

| 用途 | 推奨データ構造 | 理由 |
|------|--------------|------|
| 順次アクセスが多い | `Vec<T>` / 配列 | キャッシュ効率最高 |
| 小さな固定長集合 | `ArrayVec` / `SmallVec` | ヒープ確保なし |
| FIFO キュー | リングバッファ | キャッシュ効率 + 固定メモリ |
| 大量の短命オブジェクト | アリーナアロケータ | アロケーションコスト削減 |
| マルチスレッドキュー | ロックフリーキュー | ロック競合なし |
| メンバーシップテスト | Bloom Filter | 空間効率 |
| ゲーム・シミュレーション | SoA / ECS | SIMD・キャッシュ効率 |

## 学習目標

この章を終えると、以下のことができるようになります。

1. **CPUキャッシュの仕組み**を理解し、キャッシュミスを意識したコードを書ける
2. **`Vec<T>`の代替**（`SmallVec`、`ArrayVec`）を適切な場面で使える
3. **リングバッファ**を`unsafe`を使って実装し、その最適化手法を理解できる
4. **アリーナアロケータ**を使って大量オブジェクト生成のコストを削減できる
5. **ロックフリーキュー**の原理を理解し、`crossbeam`を安全に使える
6. **Bloom Filter**の原理と実装を理解し、空間効率の良いメンバーシップテストができる
7. **DoD（Data-Oriented Design）**の考え方でデータレイアウトを最適化できる
