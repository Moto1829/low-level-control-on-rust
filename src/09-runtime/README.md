# 第9章: Rustでのランタイム実装

## 概要

「ランタイム」とは、プログラムの実行を支える基盤コードのことです。tokioやasync-stdといったクレートは、非同期Rustの実行を可能にするランタイムを提供していますが、その内部は多くの開発者にとってブラックボックスのままです。

この章では、ランタイムを構成する核心コンポーネントをゼロから自作します。スレッドプール・タスクスケジューラ・Future/Waker・グリーンスレッド・ミニ非同期ランタイムという5つのテーマを通じて、「Rustの非同期処理はどのように動いているか」を根本から理解します。

対象は単なる async/await の使い方ではなく、その背後にある実行基盤そのものです。高水準ライブラリの API を覚えるだけでは見えない「誰がタスクを保持し、誰が再実行を決め、誰が I/O 完了を通知するのか」を、自分で部品を組み立てながら確認していきます。

---

## なぜランタイムを自作するのか

```
外部クレートを使うだけの場合:
  tokio::spawn(async { ... })  ← 何が起きているか見えない

自作ランタイムを経験すると:
  ┌─────────────────────────────────────────────┐
  │  async fn → 状態機械  → poll() 呼び出し    │
  │  Waker    → 通知      → タスク再スケジュール│
  │  epoll    → I/O完了   → Reactorが起床       │
  │  スレッド → Worker    → ジョブキューから取得│
  └─────────────────────────────────────────────┘
  すべての層が「見える」ようになる
```

自作を通じて得られる知識:

1. **デバッグ力の向上** — 非同期コードのハングや性能問題を根本から追える
2. **設計センスの獲得** — トレードオフを理解した上でライブラリを選べる
3. **低レイヤへの接続** — OS・ハードウェアとRustの橋渡しを体感できる
4. **unsafe の正当な使い方** — 本当に必要な場面での使い方が身につく

### この章で扱う 2 つの軸

本章は大きく分けて、次の 2 軸で進む。

1. **CPU 上でタスクをどう回すか**
      スレッドプールやスケジューラの話であり、どのワーカーがどのジョブを実行するか、キューをどう持つか、終了処理をどう整えるかを扱う。
2. **待機中の処理をどう再開するか**
      Future/Waker/Executor/Reactor の話であり、いま進めない処理をいったん止め、再開可能になった瞬間に効率良く poll し直す仕組みを扱う。

この 2 軸を分けて理解すると、tokio のような実運用ランタイムも読みやすくなる。たとえば work-stealing は前者の話で、epoll と Waker は後者の話である。第5節では、この両方が 1 つのランタイム内でどう噛み合うかを見る。

## 各節の内容と学習の流れ

```
[01] スレッドプール
      ↓ 「並列実行の基盤」を理解する
[02] タスクスケジューラ
      ↓ 「何をいつ実行するか」の戦略を学ぶ
[03] Future と Waker
      ↓ 「非同期の核心」— poll駆動モデルを理解する
[04] グリーンスレッド
      ↓ 「コンテキストスイッチ」をアセンブリで体感する
[05] ミニ非同期ランタイム
      ↑ すべてを統合して動くランタイムを作る
```

読み進めるときは、「第1節と第2節で実行基盤を作り、第3節で非同期の契約を理解し、第4節でコンテキスト切り替えの低レイヤを体感し、第5節でそれらを統合する」という流れを意識すると迷いにくい。特に第3節と第5節は関連が強く、`poll()` と `Waker` の動きが腹落ちしていると第5節のコードが急に読みやすくなる。

### [01-thread-pool.md](./01-thread-pool.md) — スレッドプールの自作

OSスレッドを使ったジョブキューの実装から始めます。`Arc<Mutex<Receiver>>`パターン・グレースフルシャットダウン・パニックしたワーカーの再起動など、実務でも使える設計を学びます。

**キーワード**: `JoinHandle`, `Mutex`, `mpsc::channel`, `available_parallelism`

---

### [02-task-scheduler.md](./02-task-scheduler.md) — タスクスケジューラの設計と実装

FIFO・優先度付き・work-stealingという3種類のスケジューリング戦略を比較し、`BinaryHeap`を使った優先度付きタスクキューを実装します。タスク間の依存関係（DAG）の解決にも踏み込みます。

**キーワード**: `BinaryHeap`, `crossbeam-channel`, DAG, トポロジカルソート

---

### [03-future-waker.md](./03-future-waker.md) — Futureとwaker — 非同期ランタイムの仕組み

`Future`トレイトの`poll`メソッド・`Context`・`Waker`・`RawWakerVTable`を手動実装します。`async`/`await`がコンパイラによってどのような状態機械に展開されるかを具体的なコードで確認します。

**キーワード**: `Poll`, `Waker`, `RawWakerVTable`, `Pin`, 状態機械

---

### [04-green-threads.md](./04-green-threads.md) — グリーンスレッド（コルーチン）の実装

`asm!`マクロを使ってx86-64のスタックポインタとレジスタを手動で切り替え、グリーンスレッドを実装します。スタックの確保・ガードページによるオーバーフロー検出・ラウンドロビンスケジューラまで実装します。

**キーワード**: `asm!`, `mmap`, コンテキストスイッチ, ガードページ

---

### [05-mini-async-runtime.md](./05-mini-async-runtime.md) — ミニ非同期ランタイムを作る

tokio・async-stdを一切使わずに非同期コードを動かすミニランタイムを作ります。Executor・Reactor・Wakerの3コンポーネントを実装し、TCPエコーサーバーを動かすことを目標とします。

ここでは単にサーバーが動けば終わりではなく、`Poll::Pending` がどのように再スケジュールへつながるか、Executor がなぜ I/O 完了そのものを知らなくてよいのか、Reactor がどこまで責務を持つべきかまで追います。ランタイム実装の最小構成を体験する節であり、本章全体の総まとめに相当します。

**キーワード**: Executor, Reactor, `epoll`, カスタムWaker, タイマーFuture
